- [x] `TD-010` [P1] `shared/serverapi` mixes wire DTOs with server-owned service interfaces and headless runtime execution.

  `shared/serverapi` defines request/response DTOs, validation, errors, 18 `*Service` interfaces, and concrete service behavior in one shared package.   Regression prevention must include package-boundary tests that fail when concrete service implementations, runtime handles, lifecycle orchestration, logging policy, timeout policy, or close semantics are added to `shared/serverapi`. Route-contract tests must still prove every shared request/response pair has matching remote and loopback bindings, so removing server-owned behavior from `shared/serverapi` does not reintroduce endpoint drift.

- [ ] `TD-017` [P1] Session event-log utilities still materialize unbounded histories.

  Repository guidance says no code should load full `events.jsonl` histories into memory. `server/session/store.go` exposes `ReadEvents() []Event`, which parses the entire event file and returns a slice. `server/session/snapshot.go` calls `ReadEvents` in `SnapshotFromStore`, reads all events in `SnapshotFromDir`, stores them in `Snapshot.Events`, and builds `Snapshot.Runs` from the same full slice. `server/session/fork.go` walks parent events and appends `ReplayEvent` values to an in-memory `replay` slice until the requested user-message boundary. `server/session/runs.go` implements `ReadRuns` by walking every event and accumulating all run records. `server/session/prompt_history.go` walks every event to reconstruct prompt history, and `server/sessionview/service.go` builds dormant transcript pages by scanning persisted events on demand.

  Long-running sessions can grow to gigabytes, so full event materialization creates memory pressure, slow resume/fork/snapshot operations, and higher crash risk. The presence of `ReadEvents` makes it easy for new features to accidentally violate the unbounded-history rule even when pagination or streaming projection would be sufficient.

  Remediation task: remove or quarantine full-history APIs behind test-only or explicit migration tooling. Replace production `ReadEvents` and snapshot callers with streaming projectors, bounded pagination, cursor-based readers, and domain-specific summaries. Fork replay should stream directly to the child store or spool through bounded chunks. Run listing and prompt history should use indexed metadata or incremental summaries instead of scanning entire event logs on demand. Dormant transcript reads should use bounded windows and persisted indexes where full replay is not required.

  Regression prevention must include static checks or architecture tests that fail when production code calls full-history materializers such as `ReadEvents` or stores unbounded event slices. Add large-session tests for snapshot, fork, run listing, prompt history, and dormant transcript reads that assert bounded memory behavior and cursor/pagination correctness. New persistence features must prove they do not require loading all of `events.jsonl` into memory.

- [ ] `TD-018` [P0] Boundary validation and error taxonomy remain optional conventions instead of enforced contracts.

  Validation and typed error handling vary by boundary. `shared/rpccontract/routes.go` derives `Route.ValidatesRequest` by reflecting whether the request type implements `Validate() error`, and `shared/rpccontract/routes_test.go` only checks that the metadata matches reflection. `server/transport/gateway.go` uses `decodeAndHandle` to call `Validate` only when the decoded request happens to implement that interface, while `(*Gateway).serveRunPrompt` separately decodes `serverapi.RunPromptRequest` and forwards it without transport-level validation. Request examples without route-enforced validation include `serverapi.ProcessListRequest` and `serverapi.RunPromptRequest`; `serverapi.ProcessInlineOutputRequest.Validate` requires `process_id` but does not bound `MaxChars`. Tool and model boundaries also normalize malformed inputs by convention: `server/runtime/tool_executor.go` synthesizes missing tool-call IDs, continues unknown tools as per-call error results, special-cases web-search validation, and falls back through JSON adaptation for custom tool input; `server/llm/openai_http_output_parser.go` downgrades unknown output item types to `ResponseItemTypeOther` and returns empty parsed items for function/custom tool outputs missing required IDs or names. Extensibility registration validation sometimes fails through package-init or constructor panics instead of startup diagnostics: `cli/app/commands/commands.go` panics for command names containing whitespace, `server/tools/types.go` panics for nil registries, missing definitions, and duplicate handlers, `server/tools/definitions.go` panics during `init` for malformed tool contracts, and `server/llm/provider_factory.go` panics when provider/model contracts are malformed or duplicated. Config validation is similarly source-sensitive: `shared/config/config_load.go` applies file, env, then CLI state before validation, while `shared/config/config_validation.go` gates several checks through `hasExplicitSource`, which only treats `"file"` as explicit. Error mapping follows the same convention path: `shared/protocol/jsonrpc.go` exposes only `ResponseError{Code, Message}`, `server/transport/gateway.go` maps a fixed set of sentinel errors in `protocolError`, `shared/client/remote_rpc.go` reconstructs typed client errors only for known protocol codes, and `cli/kent/main.go` emits headless run JSON errors with only coarse code and message.

  Boundary correctness depends on every route, tool, model adapter, config source, and CLI caller remembering where validation and typed errors live. Invalid zero/default values can reach service code before validation, validation can happen after dependency resolution or other side effects in route-specific handlers, model-output corruption can be flattened into opaque entries, and clients cannot reliably distinguish retryable, auth, scope, validation, not-found, unavailable, unsupported, or operator-action-required failures when a service returns an unmapped error. This increases remote/embedded divergence risk and makes boundary behavior hard to audit from one source of truth.

  Remediation task: make validation and error semantics explicit at every public boundary. The authoritative route contract must declare validation policy, accepted request invariants, and typed error set before a route can be registered; gateway unary, progress, subscription, loopback, and remote clients must use the same generated or table-driven adapter. Tool invocation and model-response adapters must reject or surface malformed IDs, unknown item types, invalid schemas, and unsupported fields through typed diagnostics instead of silent normalization. Command, tool, and provider registries should keep fail-fast contract checks, but run them through explicit startup/composition validation that returns diagnosable typed errors instead of implicit package-init panics. Config/env/CLI validation must encode source-aware invariants explicitly for file, env, CLI, and generated defaults. Replace ad hoc sentinel checks with an error taxonomy that carries typed code, public message, retry/action policy, and optional structured details without leaking internal error strings.

  Regression prevention must include route-contract tests that fail when a request type with fields has no validation policy, when progress/subscription routes bypass validation, when request validation happens only in service code, or when a service can return a public error without a route-declared protocol mapping. Add tool/model adapter tests for missing IDs, unknown output item types, malformed custom tool input, invalid schemas, and malformed registry/provider contracts, plus config tests proving file/env/CLI source-specific invariants are enforced uniformly. Startup validation tests must prove extension mistakes surface as typed composition failures rather than package initialization panics. Client and CLI tests must prove known errors map to typed errors and unknown internal failures preserve diagnostic text only in the appropriate operator-visible channel.

- [ ] `TD-019` [P0] Runtime, session transcript, and metadata authorities do not share one commit/recovery boundary.

  Runtime and session persistence use several partial commit boundaries. `server/runtime/engine_message_ops.go` mutates `e.chat` before calling `e.store.AppendEvent` in `appendMessage` and `appendMessageWithoutConversationUpdate`; if persistence fails, live runtime state has already changed. `server/runtime/compaction_utils.go` mutates the chat store through `replaceHistory` before appending the `history_replaced` event, then only updates related store metadata such as compaction reminders and usage state after a successful append. `server/session/store.go` names `appendEventsAtomicLocked`, but the implementation performs separate operations: `appendEventsLogLocked` appends to `events.jsonl`, then `s.meta.LastSequence`, `UpdatedAt`, and counters are mutated, then `persistMetaLocked` writes `session.json.tmp` and renames it to `session.json`. If metadata persistence fails after the append, the event log may contain events whose associated metadata fields, first-prompt preview, freshness-derived metadata, or observer snapshot were not durably committed. The post-migration metadata layer adds another authority: `server/metadata/store.go` installs itself as a `session.PersistedSessionResolver`, reconstructs `session.Meta` from SQLite rows in `sessionMetaFromRecordRow`, and upserts `session.PersistedStoreSnapshot` into SQLite in `upsertSessionSnapshot` while large artifacts such as `events.jsonl` remain file-backed under `artifact_relpath`. Startup repairs only part of this split state: `server/session/event_log.go` reconciles `Meta.LastSequence` and conversation freshness from the log in `bootstrapEventLogStateLocked`, but other metadata derived before the append is not rebuilt there. The same persistence layer rewrites `events.jsonl` during periodic compaction through `readEventsFile` and `writeEventsFile`, and `writeEventsFile` replaces the log without fsyncing the file or parent directory.

  A crash, disk error, permission error, failed append, failed SQLite update, failed artifact move, or failed rename can leave live runtime state, durable event rows, file-backed session artifacts, and SQLite-derived metadata in different states. Users can see transient transcript entries, stale compaction projections, missing prompt previews, stale run/session summaries, stale workspace/session targeting, or incomplete cache-warning visibility after restart even though some part of the write path appeared to succeed. Engineers must reason about recovery behavior separately for each live projection, file artifact, SQL row, and derived metadata field instead of relying on one commit boundary.

  Remediation task: define one runtime/session persistence commit contract for transcript mutations, event appends, file-backed artifacts, SQLite metadata rows, and derived metadata. Runtime chat state should be updated only from successful durable commits, or live mutations must have explicit rollback/replay semantics that are covered by tests. Either make the event log the sole durable authority and rebuild all derived metadata/indexes idempotently on startup, or introduce a transactional metadata/index authority that commits event rows, artifact references, and metadata updates together. The implementation must respect existing decisions that normal event writes are append-only and fsync policy is configurable, while making failure semantics explicit for no-fsync modes. Event-log compaction and SQLite snapshot import must use crash-safe replace/update protocols with temp-file cleanup, directory sync where configured, and startup diagnostics for interrupted rewrites or mismatched SQL/file state.

  Regression prevention must include injected-failure and crash-restart tests for failures after live chat mutation but before event append, after event append but before file metadata persistence, after temp metadata write but before rename, after SQLite metadata update but before artifact commit, after artifact commit but before SQLite update, during event-log compaction rewrite, and during observer publication. Tests must prove that live and replayed sessions agree on transcript entries, compaction projections, last sequence, first prompt preview, conversation freshness, workspace/session targeting, run summaries, goal state, and cache-warning visibility, or surface a typed corruption diagnostic. Future persistence code must not use "atomic" naming unless tests prove all live and durable resources in the write are committed, rolled back, or recoverable as one unit.

- [ ] `TD-020` [P1] Background shell output streaming has no bounded chunk or backpressure contract.

  Background shell output is retained in temp log files and streamed through ad hoc subscription reads. `server/tools/shell/manager.go` starts every command with an app-global `Manager`, writes stdout/stderr into `processEntry.logFile`, keeps only a bounded in-memory `pendingOutput`, and emits completion events through one non-blocking `notify` channel. `server/tools/shell/manager_output.go` implements `(*outputSubscription).Next` by opening the process log, seeking to `offset`, and calling `io.ReadAll(file)` for all bytes currently available. `server/transport/gateway.go` forwards each returned `ProcessOutputChunk` as one JSON-RPC notification in `serveProcessOutputSubscription`, and `shared/client/remote_stream.go` unmarshals each notification as one chunk. `shared/serverapi/process_output.go` validates only `process_id` and non-negative `offset_bytes`; there is no negotiated max chunk size, server-side page size, subscriber buffer policy, or slow-subscriber behavior in the contract. `server/tools/shell/manager.go` also creates a temp directory with `os.MkdirTemp("", backgroundLogDirPrefix)`, while `(*Manager).Close` kills processes and waits but does not remove the temp log directory.

  A process that writes a large burst can force one subscription poll to read and marshal a large tail into memory, block a transport send on a huge notification, or make clients process output without a predictable paging budget. Completion notices and inline output use truncation, but the streaming contract itself is unbounded. Temp logs can also accumulate for the lifetime of the process and remain after manager close, increasing disk usage during long-running agent sessions.

  Remediation task: define a bounded background-output stream contract. The process-output API must require a server-enforced maximum bytes/chars per chunk, return `next_offset_bytes`, expose truncation/backpressure metadata, and let clients resume from offsets without ever reading the whole remaining log. The manager should retain log files under an explicit ownership and retention policy, remove or archive its temp directory on close according to that policy, and make completion events use the same bounded reader. Keep the v1 product decision that background process ids are app-global and owner session metadata is advisory, but make output memory/disk lifetime explicit and testable.

  Regression prevention must include large-output tests that subscribe from offset zero and from a tail offset while asserting bounded memory/chunk size, correct offset progression, completion delivery, slow-subscriber behavior, and cleanup of manager temp logs on close. Transport tests must fail if one process-output notification can exceed the declared chunk limit or if inline/completion/streaming output paths use different truncation and offset semantics.

- [ ] `TD-022` [P1] `server/session.Store` holds its global mutex across filesystem scans, rewrites, and fsync work.

  `server/session/store.go` uses one `sync.Mutex` to protect metadata, event-log offsets, freshness, and persistence state. Many methods hold that mutex while doing filesystem work. `SetName`, `SetInputDraft`, `SetCompactionSoonReminderIssued`, and `SetWorktreeReminderState` call `persistMetaLocked`, which can `EnsureDurable`, marshal metadata, write `session.json.tmp`, and rename it while the store lock is held. `AppendEvent`, `AppendTurnAtomic`, and `AppendReplayEvents` hold the same lock through `appendEventsAtomicLocked`, which can compact the event log, append to `events.jsonl`, fsync according to policy, mutate metadata, and persist metadata. `ReadEvents` and `WalkEvents` hold the lock while opening and parsing the whole event log; `server/session/event_log.go` performs `readEventsFile`, `writeEventsFile`, `walkEventsFile`, tail repair, and compaction under callers' locked paths.

  The session store is a central runtime persistence boundary, so holding one global lock across disk I/O, full-log scans, compaction rewrites, and fsync can stall unrelated session operations, runtime reads, draft persistence, run metadata, goal updates, fork/replay, and live/dormant projections. Large sessions amplify the problem because the longest operations are unbounded history reads or full rewrites. This also makes lock-order review harder because callers cannot distinguish quick in-memory critical sections from potentially slow filesystem work.

  Remediation task: separate in-memory state protection from persistence I/O. Introduce a small critical section that builds immutable write/read plans, then perform filesystem operations outside the store mutex with explicit version checks or a serialized persistence worker. Event-log scans and transcript projections should use streaming readers that do not block unrelated metadata mutations. Metadata persistence should commit through a narrow write transaction that validates the expected store version before publishing results. Preserve current ordering and crash-recovery semantics while making lock ownership and I/O boundaries explicit.

  Regression prevention must include concurrency tests that run draft persistence, event appends, read-model queries, prompt-history reads, and compaction/fork operations against slow or blocking filesystem fakes and prove unrelated operations are not serialized behind long scans. Add lock-scope tests or static checks that fail when `server/session.Store.mu` is held while calling file open/read/write/rename/fsync or full-history projector functions.
