# Tech Debt Findings Catalog

This catalog locks in verified debt findings. Use `docs/dev/techdebt/techdebt_criteria.md` as the research checklist; add entries here only after collecting concrete evidence.

Each entry is checklist-shaped so remediation can be tracked without inventing a separate issue system. These are implementation tasks, not architectural essays: keep findings concise, concrete, checkable, and tied to regression prevention.

Entry shape: checklist title line, summary evidence paragraph, impact paragraph, remediation task paragraph, regression-prevention paragraph.

## P1 Findings

- [x] `TD-001` [P1] RPC endpoint contract is hand-mirrored across protocol, DTOs, client, gateway, and tests.

  The same endpoint facts are repeated across many files. `shared/protocol/handshake.go` contains method constants such as `MethodRuntimeSubmitUserMessage`. `shared/serverapi/runtime_control.go` contains `RuntimeSubmitUserMessageRequest`, `RuntimeSubmitUserMessageResponse`, and `RuntimeSubmitUserMessageRequest.Validate`. `shared/client/remote.go` contains remote wrappers such as `(*Remote).SubmitUserMessage`. `server/transport/gateway.go` contains server dispatch in `(*Gateway).dispatch`. `shared/protocol/handshake.go` also contains `allowedPreAuthMethods`, consumed by `(*Gateway).methodRequiresServerAuth`. Tests in `shared/client/remote_transport_test.go`, `server/transport/gateway_test.go`, and `server/transport/gateway_part2_test.go` manually reference the same route family.

  Adding or changing one endpoint requires synchronized edits across unrelated files. The compiler does not prove route registration, auth policy, scope policy, subscription semantics, or remote/loopback parity. Failure modes include runtime `method not found`, missing pre-auth access, missing authorization checks, wrong connection strategy, or remote-only failure hidden by loopback tests.

  Remediation task: replace the hand-mirrored RPC contract with one authoritative route contract. The contract must define the method name, request type, response type, validation behavior, auth policy, scope policy, route kind, stream semantics, connection strategy, and handler dependency for every RPC method. Gateway dispatch, pre-auth method calculation, subscription registration, and route coverage tests must be derived from this contract instead of independently maintained switch/list logic. Remote and loopback clients must either be generated from the same contract or covered by compile-time/runtime parity tests proving that every route has matching client bindings and identical request/response types.

  Regression prevention must include tests that fail when a protocol method is missing a route contract, when a route lacks auth or scope policy, when pre-auth exposure drifts from the route policy, when unary/subscription route kind is wrong, when a client wrapper is missing, or when loopback and remote behavior diverge. Adding a new endpoint should require one source-of-truth route declaration plus its service implementation and endpoint-specific tests, not scattered edits across protocol, gateway, client, and policy lists.

Routes that remain volatile must have contract tests proving clients do not retry them as idempotent operations and docs that describe retry behavior.
